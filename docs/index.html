the five stages

Every dispatch runs through the same named stages.

Lifted from Arjen's Specialist Builder — the workflow naming is good, no reason to reinvent it. Brief-file dispatch, durable artifacts, and Pages-by-default output are ours.

stage_01

CALIBRATE

Reads brief. Surfaces preconditions as questions. Mandatory, not skippable. STOPS until operator answers.

stage_02

PLAN

Surfaces closest analog, folder tree, key patterns. You sign off or redirect. No files written yet.

stage_03

BUILD

Produces files. Cites source repos and research reports. Ships docs/index.html.

stage_04

VERIFY

Runs 13-row ICM checklist. Pages self-contained discipline check. Reports pass/fail per row.

stage_05

HAND OFF

Outputs gh CLI deployment commands. Updates STATUS. Appends upstream breadcrumb. Worker doesn't deploy by default.

what makes this not a clone

Three flair axes Arjen and Virgilio don't have.

Specialist Foundry sits in the empty slot between Arjen's developer-grade chat-prompt and Virgilio's non-technical wizard. Operator-grade buyer; durable artifacts; integrity as a feature.

axis_01

Audit-trail dispatch

Every build has a brief file at briefs/<date>-<slug>.md, optional question files surfacing preconditions, and a STATUS update afterward.

Read a year-old build's brief + question file + STATUS and reconstruct what was decided and why. Chat-prompt and wizard dispatches are ephemeral. Foundry's is durable.

axis_02

Pages-by-default

Every Foundry-built specialist ships with a self-contained docs/index.html — inline CSS, no external dependencies, with its own visual identity.

Push the repo + enable Pages from /docs → public URL at <username>.github.io/<slug>/ in 60 seconds. Distinct aesthetic per build is refusal-grade discipline — no template clones.

axis_03

Integrity-guardrails

Refuses to manufacture unlabeled testimonials, refuses high-harm domains (medical / legal-licensed / suicide-intervention), enforces empty-region rules.

If you want fictional testimonials before cold-test, Foundry insists on inline (illustrative composite, fictional) labels. Tested on a bereavement-product build where this actually mattered.

setup

About 5 minutes, one time.

If you can use Claude in a web browser, you can set this up. No install, no API key, no monthly cost beyond your Claude subscription.

Clone or download the repo

From GitHub: git clone the repo, or click the green Code button and download the ZIP. You only need the markdown files and the reference folder.

git clone https://github.com/NFTYoginis/specialist-foundry.git

Create a new Claude Project

Go to claude.ai/projects, click New Project, name it "Specialist Foundry."

Upload as Project Knowledge

Upload identity.md, rules.md, examples.md, and all files under reference/. Drag and drop works.

Set up your working folder

Create a working folder somewhere on disk with two sub-folders: briefs/ and builds/. Foundry writes to both.

mkdir -p ~/my-specialists/briefs ~/my-specialists/builds

Write your first brief and dispatch

Write a brief into briefs/<date>-<slug>.md describing the specialist you want built (target buyer, domain, output shape, scope limits). The brief template is in this repo's examples.md.

Then paste this verbatim into your Claude project chat:

Read the brief at briefs/<filename> and dispatch through the five Foundry stages.

Foundry runs CALIBRATE → PLAN → BUILD → VERIFY → HAND OFF.

faq

Common questions.

Why brief-file dispatch instead of a chat prompt?

The brief file is the audit trail. Without it, your dispatch is ephemeral — when the chat closes, you can't reconstruct what you decided or why. A year from now, you can read the brief + the question file + the STATUS line and rebuild context in 90 seconds. This matters when you ship more than one specialist or work on a team.

Arjen's Specialist Builder uses a chat prompt and is 10 minutes per specialist — faster, less ceremony, ideal for one-off builds by a developer comfortable with the inline-markdown output. Foundry is slower (~30 min) and adds the audit-trail overhead deliberately.

What's the actual difference from Arjen's Specialist Builder?
  • Dispatch: brief file (durable) vs. chat prompt (ephemeral)
  • Output: pre-built folder on disk (you don't copy-paste) vs. inline markdown code blocks
  • Pages-ready landing page: always shipped (docs/index.html) vs. not included
  • Integrity-guardrails: three explicit gates (launch-layer, high-harm-domain, unlabeled-fake-proof) vs. high-harm-domain only
  • Workflow stages: same 5 names (lifted from Arjen — credit given in this repo's README and acknowledgments)

If you're a developer who wants one specialist fast and you're comfortable copy-pasting markdown, Arjen's is the better fit. If you ship multiple specialists, want Pages-by-default, and care about the audit trail, Foundry's is.

What's the actual difference from Virgilio's ICM Workspace Builder?
  • UX: brief file (technical) vs. wizard with archetype selection (friendly for non-technical)
  • Architecture layers: ICM-canonical 5 + Pages-by-default vs. 6-layer system (Virgilio's adds CLAUDE.md, CONTEXT.md, stage contracts, _config/, output/, optional skill-starters)
  • Output medium: pre-built folder vs. downloadable .zip
  • Buyer: operator-grade (technically comfortable) vs. non-technical archetype-picker

Virgilio's architecture is materially richer than Foundry's. If you want CONTEXT.md + stage contracts in your output, Virgilio's is the better fit. Foundry kept the ICM-canonical 5 deliberately to stay close to the source repos and minimize per-build setup time.

Why does each Foundry-built page look visually different?

"Pages-by-default" is a real flair only if each built specialist looks like its own artifact, not a template clone. The pattern doc reference/patterns/pages-by-default.md requires that every build propose a distinct visual identity at PLAN stage — palette, type system, density, components, hero treatment. Same template wearing different paint fails VERIFY.

This page uses an operator-tooling aesthetic (dark slate + monospace section labels + sharp corners) because Foundry is itself a tool. The funeral compass uses a bereavement-grade aesthetic (warm white + sage accent + serif headlines + spacious layout) because its domain calls for dignity, not promotional warmth. Both ship under the same Foundry discipline.

Can I fork this and use a different workflow / different gates?

Yes — MIT-licensed. The architecture is meant to fork. If your fork makes substantial workflow or gate changes, please rename it ("Specialist Foundry — Acme Edition" / "Compliance Foundry" / whatever) so the upstream lineage stays clear.

If you build something that should go upstream, file a PR. The two refusal gates that don't bend are unlabeled-fake-proof and high-harm-domain — these are integrity gates, not features. Forks that strip them lose the lineage; that's your call, but I won't merge it back.

What if my brief is for a high-harm domain (medical, legal, suicide)?

Foundry refuses at CALIBRATE and points you to certified professionals:

  • Medical diagnosis → licensed physicians; for second opinions, Cleveland Clinic Express Care, Teladoc, or your insurer's nurse line
  • Legal advice in licensed jurisdictions → an attorney licensed in the relevant state/country; your state or country bar association has a referral line
  • Suicide intervention → 988 Suicide and Crisis Lifeline (US, call or text 988); for other countries, International Association for Suicide Prevention

Adjacent domains (medical TRIAGE for non-emergencies, legal admin, grief admin) are in scope — see the Online Funeral Family Compass for the bereavement adjacent case.

Do I have to use GitHub Pages?

No — the docs/index.html file is self-contained and works via file:// protocol. You can email it, archive it, host it on Netlify, or just keep it for local preview. But Pages is the path of least resistance: push the repo, enable Pages, public URL in 60 seconds.

Why insist on labeled-illustrative testimonials? Can't I just use real ones?

Yes, if they're real. The refusal is on unlabeled fictional ones. Real testimonials with provenance are always preferred — operator supplies them, attributes accurately, the person has consented. Foundry doesn't second-guess provenance.

The labeled-illustrative discipline exists for the pre-launch period, when you don't have real testimonials yet and you want the landing page to communicate scenarios anyway. Inline labels keep that honest.

What does it cost?

The Foundry itself is free, MIT-licensed. The cost is your Claude subscription (free tier exists; paid tier varies). There is no third-party SaaS, no API key, no recurring fee added by this Foundry. Each Foundry-built specialist is also MIT-licensed unless your brief specifies otherwise.

How is this related to Realtor Copilot v2 and the Grief Admin Compass?

Both of those are built specialists, not meta-builders. They are example outputs of the architecture Foundry codifies. Realtor Copilot v2's empty-region rule and Pages-ready landing page pattern were lifted into Foundry's defaults. Grief Admin Compass's 4-column triage shape and admin-first voice informed how the first Foundry build (the Online Funeral Family Compass) was structured. Acknowledgment to both at the bottom of this Foundry's README.

I built something with Specialist Foundry. How do I list it back?

Open an issue with the repo URL + a one-line description. Specialists built with Foundry get listed in this repo's README. The cousin lineage matters and we keep it visible.
